## W3.0a — Fake hardening: fault-injector attempt salt

`FaultInjector::decide` is a pure function of `(seed, target,
operation, request_identity, channel)` — a retried request with
identical bytes draws the identical terminal fault forever. Ops whose
identity is attempt-invariant (restore_snapshot, create_vm,
score_batch — where the *same* `client_batch_id` is the point — and
the unit-struct requests) would permanently poison ~5% of identities
and make W3.6's retry-equivalence test unpassable as designed.

- Extend `FaultInjector` with a per-`(target, operation)` attempt
  counter (`Cell<u64>` folded into the draw), so retries draw fresh
  outcomes while whole runs stay seed-deterministic.
- Existing `orch-fakes` contract tests updated for the new derivation;
  transcript-stability test for the disabled-plan path unchanged.

**Done when:** workspace green on one tonic; `orch-sched` compiles with
adapter unit tests (latency mapping incl. sleep placement, error
passthrough, timeout charging); fault salt landed with contract tests.

## W3.1 — `SlotView` (`orch-sched/src/slots.rs`)

- Seed from `ListSlots` at startup; maintain via `WatchSlots` drains
  (the fake's incremental cursor model maps to a periodic virtual-time
  drain task; the real impl at M6 swaps in a server-stream).
- Free-slot accounting per endpoint; `acquire(class) -> HostPick`
  suspends until a class-compatible slot is likely free;
  `RESOURCE_EXHAUSTED` on the actual RPC is treated as backpressure
  (release-and-retry), never as an error.
- Determinism-class gating: parent node's `determinism_class` (from
  attrs) vs `GetWorkerInfo().class`; `allow_class_mismatch` (default
  false) refuses dispatch on mismatch with a distinct, grep-able
  FAILED-reason string (M5 forward-design).

**Accept mapping:** feeds the utilization bar (slots never idle while
work exists) and the shrink/grow bar (WatchSlots transitions shrink and
grow the free view mid-run).

## W3.2 — Worker driver (`orch-sched/src/driver.rs`)

- `run_job(JobSpec) -> JobResult`: `RestoreSnapshot|Fork → InjectInputs
  → Run(frame_budget, hard_icount_cap) → TakeSnapshot(seal, CaptureSpec)
  → DestroyVm`, per API.md §2.2. Entropy seed = `derive(seed,
  "entropy", batch_seq, job_idx)`; bootstrap uses `derive(seed,
  "boot")` (API.md §2.1 step 1) — extend `orch-core::rng` with **both**
  purpose tags, golden-vector tested.
- **Fork discipline (review finding):** the fake (like a real worker)
  requires a Paused parent and freezes it per fork; K independent
  per-job forks of one shared parent would collide
  (`FailedPrecondition` for jobs 2..K). So: sibling forks of one
  parent are batched into a **single** `ForkRequest` at dispatch time
  (its `entropy_seeds: Vec` exists for exactly this), Restore is used
  otherwise, **retries always take the Restore path** (never re-Fork a
  frozen parent), and frozen-parent slots count as *busy* in
  utilization accounting.
- Burst→`ScheduledEvents` translation: PadBurst segments to PadSet
  events at **absolute** `at_frame`; the fake (and a frame-coherent
  real worker) rejects `frame <= current frame_counter`, so the base is
  parent `frame_counter` **+ 1** + cumulative segment offset —
  golden-tested against the strict-future validation.
- Verdict mapping table (API.md §2.2): BUDGET_REACHED/GOAL_SATISFIED/
  NEXT_SDK_EVENT → ok; HARD_CAP → guest_hang; GUEST_HALTED/FAULTED →
  guest_crash. `guest_hang` children carry the frontier-ineligible
  marking into commit.
- `bootstrap(BootstrapSpec) -> RootCapture`: `CreateVm → Run(until
  Ready sdk event, bootstrap icount cap) → TakeSnapshot → DestroyVm`
  (API.md §2.1); scoring + root CreateNode stay in the M4 runner —
  the driver returns the capture only.
- `JobResult` carries child ref, input_log_id, the three digests,
  feature_bytes, fb_lz4, fb_info (frame_counter), verdict, sdk events.

**Accept mapping:** purity of the composition is what W3.4's retry
equivalence test leans on; bootstrap feeds M4's autonomy bar.

## W3.3 — Pipeline (`orch-sched/src/pipeline.rs`)

- Stages per ARCHITECTURE.md §6.2: S (select+synth, done by the caller
  — the pipeline receives ready `Batch{seq, node, bursts}`) → bounded
  queue (cap `max_inflight_batches − 1`) → E (K× `run_job` fanned out,
  bounded by SlotView) → bounded queue (cap 1) → C handoff
  (`next_completed(mode)`).
- `next_completed` semantics: FAST = completion order; DETERMINISTIC =
  strictly `seq` order with job results sorted by job index
  (`max_inflight_batches` already coerced to 1 by config validation).
- Queue depths + slot busy/free exported as plain atomic gauges behind
  a small `Gauges` struct (Prometheus wiring is M4's server; names
  fixed now: `orch_pipeline_queue_depth{stage}`,
  `orch_slot_utilization`, `orch_jobs_failed_total`,
  `orch_batch_latency_seconds{stage}`).
- All channels `tokio::sync::mpsc::channel(cap)`; a full channel
  suspends the upstream stage (backpressure bar).
- No unbiased `select!` anywhere in the pipeline; fixed spawn order
  (D2 determinism footguns).

## W3.4 — Retry (`orch-sched/src/retry.rs`)

Per ARCHITECTURE.md §6.4, config-driven from `SchedulingConfig`:

- Driver jobs: timeout `job_timeout_s` (virtual time in tests); on
  timeout/transport error/worker-crash status → best-effort DestroyVm →
  exp backoff (base `retry_backoff_ms`, ×2, cap 10 s) → full re-run on
  any free class-compatible slot; up to `retry_max`. After `retry_max`:
  FAST abandons the job (batch continues, `orch_jobs_failed_total`++,
  journal the gap); DETERMINISTIC aborts the experiment FAILED (reason
  string fixed + grep-able).
- Synth/store RPCs: same backoff; ProposeBursts retried behind the
  existing fingerprint guard (`propose_bursts_with_fingerprint_guard`);
  CreateNode blind-retry is safe (idempotent); PutMetadata retries
  re-read the generation.
- ScoreBatch: same backoff; the retry license is `client_batch_id`
  (`"b<seq>"`) dedup replay, verified against FakeScorer's batch cache.
